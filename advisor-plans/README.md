## Findings considered and rejected

Recorded so they aren't re-audited next run. Each was opened during the audit
and closed during vet by reading the actual code.

- **".env with live credentials is committed to git" (flagged HIGH/security)**:
  **FALSE.** `.env` is untracked, never in git history, and explicitly ignored
  (`.gitignore` has `.env` + `.env.*`). The subagent assumed it was committed
  without running `git ls-files`. No action. (Plan 004 adds the *example* file,
  which is the only `.env*` that should be tracked.)
- **"OAuth callback session fixation" (flagged HIGH/security)**: **FALSE.**
  `controller/src/auth/routes.rs:78-90` validates an encrypted, browser-bound
  cookie + one-time CSRF nonce + PKCE — the standard, correct OAuth flow. The
  attacker scenario requires the victim's encrypted cookie. No action.
- **Slot / port / group-cap concurrency races (flagged MED/correctness)**:
  **VERIFIED SAFE.** All three write paths lock their contended rows
  `FOR UPDATE` before counting occupants: single slot
  (`deployments.rs:457`), per-server port pool (`deployments.rs:705`), and
  group members (`gpu_groups.rs:50-74`, deliberately locking the member rows
  then counting). The locking design is sound. No action.
- **Local-login CSRF (flagged MED/security)**: **DOWNGRADED, not planned.**
  The session cookie is `HttpOnly + Secure + SameSite=Lax`
  (`auth/cookies.rs:21-23`), which blunts login-CSRF; impact on an admin login
  endpoint is low. Revisit only if the auth surface widens.
- **OAuth 304 handling / alternate-localhost URL forms / session-sweeper timing
  (LOW/security)**: speculative, low-signal; not worth a plan. Recorded so
  they're not re-surfaced.
- **`deployments.rs` is 1139 lines (tech-debt)**: **NOT a finding** per
  doctrine — judged by responsibility cohesion, not line count. Its functions
  are all deployment-lifecycle concerns. Revisit only if it becomes a churn
  hotspot.
