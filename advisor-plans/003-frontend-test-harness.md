## Why this matters

The frontend (~9K lines, 9 pages) is the operator's only interface to a system
that starts and stops GPU workloads, and it has **zero automated tests** — no
test runner, no test files, nothing in `scripts/check.sh`'s frontend gate
beyond `lint` + `build`. `docs/TESTING.md` § Frontend already admits component
tests are aspirational. The load-bearing pure logic — the single state→color
map (a Project Invariant: "semantic color tokens only ... one state→color map")
and the slot-occupancy/eligibility math that drives the deploy drag-and-drop —
has no regression net, so a refactor can silently flip a slot's deployability
or a status color and nothing catches it before the operator does.

This plan installs Vitest + Testing Library, wires it into the verification
gate, and seeds it with high-value tests on the **pure** modules (no DOM, no
network) so the harness proves out before anyone writes component tests.

## Steps

### Step 1: Install Vitest + Testing Library and add scripts

Install the devDeps (command above). Add to `frontend/package.json` scripts:

```json
"test": "vitest",
"test:run": "vitest run"
```

**Verify**: `cd frontend && npx vitest --version` → prints a version; exit 0.

### Step 2: Configure the test environment

Add Vitest config (`jsdom` environment, globals on). If `vite.config.ts`
already imports `defineConfig` from `vite`, add a `test: { environment:
'jsdom', globals: true }` block (you may need
`/// <reference types="vitest/config" />`). Otherwise create
`frontend/vitest.config.ts`. Match the existing path-alias setup (`@/`) so
imports resolve.

**Verify**: a trivial `expect(1+1).toBe(2)` test in `src/lib/states.test.ts`
runs green: `cd frontend && npm run test:run`.

### Step 3: Seed real tests on `states.ts`

Test the state→color/label map: for every state enum value the map claims to
cover, assert it returns a defined token (no `undefined`), and assert the
specific invariant cases the UI relies on (e.g. a Running slot vs a Freeing
slot map to different tokens). Read `frontend/src/lib/states.ts` first and
assert against what it actually exports — do not invent states.

**Verify**: `cd frontend && npm run test:run` → states tests pass.

### Step 4: Seed real tests on `slots.ts`

Test the slot occupancy + deploy-eligibility functions: an empty slot is
deployable; a slot at `max_occupants` is not; a slot held by an external
(unmanaged) container reports correctly. Use plain object fixtures matching the
function signatures — read `slots.ts` to get the exact input types.

**Verify**: `cd frontend && npm run test:run` → all pass; ≥ ~6 assertions
across the two files.

### Step 5: Wire tests into the verification gate

In `scripts/check.sh`, extend the frontend block to run tests:

```bash
if [ -f frontend/package.json ]; then
  (cd frontend && npm run lint && npm run test:run && npm run build)
fi
```

Update `docs/TESTING.md` § Frontend: replace the "manual until visual testing
is added" note with the now-true state — Vitest harness exists, pure `lib/`
logic is covered, component/visual tests remain the open item.

**Verify**: `bash scripts/check.sh` → "check.sh: all gates passed".

## Test plan

The seed tests ARE the deliverable. Minimum bar:
- `states.test.ts`: every mapped state resolves to a defined color + label;
  two distinct states don't collide on the same token where the UI depends on
  them differing.
- `slots.test.ts`: free / at-capacity / externally-held cases for the
  eligibility function.
- Verification: `cd frontend && npm run test:run` → all pass.

## STOP conditions

- A target `lib/` function can't be imported in isolation (pulls in a DOM or
  network side effect on import) — report it; do not refactor app code here.
- `vitest` and the installed React 19 / Vite version conflict on peer deps —
  report the conflict rather than forcing `--legacy-peer-deps` blindly.
- The build starts failing after config changes (Vitest config bleeding into
  the Vite build) — separate the configs and report if it persists.
